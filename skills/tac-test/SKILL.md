# TAC Test — TDD Enforcement

TAC follows strict Test-Driven Development. Every piece of code goes through RED-GREEN-REFACTOR.

## The Iron Law

```
NO CODE WITHOUT A FAILING TEST FIRST
```

This is not optional. This is not "when convenient." Every feature, every bugfix, every change.

## TDD Cycle Per Task

```
RED    → Write the test. Run it. Watch it FAIL.
GREEN  → Write MINIMAL code to make the test pass. Run it. Watch it PASS.
REFACTOR → Clean up. Run tests. Still PASS.
COMMIT → git add + commit
```

## How TAC Enforces TDD

### During DESIGN Stage

The plan generated by `/tac-design` MUST include test files for every task:

```
Task 1: Payment models
  Files:
    - Create: inventree_patches/payments/models.py
    - Test: tests/test_payments_models.py    ← REQUIRED
  
  Steps:
    1. Write test_payment_model_creation()    ← RED
    2. Run pytest — verify FAIL               ← Watch it fail
    3. Write Payment model                    ← GREEN
    4. Run pytest — verify PASS               ← Watch it pass
    5. Commit                                 ← Lock it in
```

If a plan has tasks without test files, the DESIGN stage adds them.

### During AUTO Stage (auto-spawn waves)

Each spawned agent follows TDD order:

```
Agent receives task:
  1. Write test file FIRST
  2. Verify test fails (run pytest or appropriate test runner)
  3. Write implementation
  4. Verify test passes
  5. Report: "RED confirmed, GREEN confirmed"
```

If an agent writes code before tests, that's a TAC violation. The safety check catches it.

### During SAFE Stage

`/tac-safe` verifies:
- Every new .py file has a corresponding test file
- Tests actually run (not just exist)
- No untested code paths in critical areas

## Stack-Aware Testing

| Stack | Test Runner | Test Location | Pattern |
|-------|------------|---------------|---------|
| django-ims | pytest | tests/ | test_{module}.py |
| django-fresh | pytest | tests/ | test_{app}_{feature}.py |
| react-full | vitest/jest | src/**/*.test.tsx | {Component}.test.tsx |
| python-api | pytest | tests/ | test_{module}.py |

Read from `.tac/stacks/{stack}.json` for project-specific test conventions.

## Test Categories

### Unit Tests (Wave 1-2 agents write these)
- Model creation, field validation
- Serializer input/output
- Pure function logic
- No DB, no network

### Integration Tests (Wave 3 agents write these)
- API endpoint responses
- View rendering
- Multi-model interactions

### Deploy Tests (Wave 4 — sequential)
- File manifest validation
- Migration chain verification
- Service name verification

## Red Flags — STOP

These mean TDD was violated:

- Code file exists without corresponding test file
- Test was written AFTER implementation
- "I'll add tests later" — NO. Tests first. Always.
- "This is too simple to test" — Simple code breaks. Test it.
- "Tests slow me down" — Tests prevent rollbacks. They save time.

## Commands

- `/tac-test` — Run all tests for current feature
- `/tac-test --coverage` — Show test coverage for feature files
- Auto-invoked during AUTO stage — no need to call manually

## Anti-Hallucination

- NEVER generate test files that import non-existent modules
- Read models.py before writing test assertions
- Read urls.py before testing API endpoints
- If test runner isn't installed, flag it — don't skip tests
